Git Branch Management Guide for Your Web App

This guide outlines how to manage your Git branches for handling minor fixes on the live app (deployed from main) and developing new features or large improvements on the development branch. The hotfix branch is used for quick, minor fixes to avoid disrupting ongoing development work.

Assumptions:

  • Your remote repository is named origin (adjust if different).
  • Render auto-deploys from main, so pushes to main trigger deployments.
  • development is already up-to-date with main.
  • Always pull the latest changes before starting work to avoid conflicts.

1. Handling Minor Fixes for the Live App (Using Hotfix Branch)

For small bug fixes or minor updates that need to go live quickly without waiting for the full development cycle:

Step-by-Step Process:

  1. Update your local main branch: Ensure you're starting from the latest production code.
    2. git checkout main  # Switches to the main branch locally
git pull origin main  # Fetches and merges the latest changes from the remote main branch
  2. Create or switch to the hotfix branch: If the hotfix branch doesn't exist, create it from main. If it does, update it.
    3. git checkout -b hotfix  # Creates and switches to a new hotfix branch from main
# OR
git checkout hotfix  # Switches to the existing hotfix branch
git pull origin hotfix  # Fetches and merges the latest changes from the remote hotfix branch
  3. Make your changes: Edit files and test locally.
  4. Commit your changes: Save your changes with a descriptive message.
    5. git add .  # Stages all modified and new files for commit
git commit -m "Fix: [brief description of the fix]"  # Commits staged changes with a message
  5. Push the hotfix branch to remote: (Optional, for collaboration or backup).
    6. git push origin hotfix  # Pushes the local hotfix branch to the remote repository
  6. Merge hotfix into main: This will deploy the fix via Render.
    7. git checkout main  # Switches back to the main branch
git merge hotfix  # Merges changes from hotfix into main
git push origin main  # Pushes updated main branch to remote, triggering Render deployment
  7. Merge hotfix into development: To keep development in sync and avoid future merge conflicts.
    8. git checkout development  # Switches to the development branch
git pull origin development  # Fetches and merges the latest changes from remote development
git merge hotfix  # Merges hotfix changes into development
git push origin development  # Pushes updated development branch to remote
  8. Delete or reset hotfix if needed: After merging, you can delete the branch or keep it for future hotfixes.
    9. git branch -d hotfix  # Deletes the local hotfix branch
git push origin --delete hotfix  # Deletes the remote hotfix branch

Best Practices for Hotfixes:

  • Keep changes small and isolated to minimize risks.
  • Test thoroughly locally before merging to main.
  • If conflicts arise during merge, resolve them carefully.

2. Developing New Features or Large Improvements (Using Development Branch)

For adding new features, refactoring, or major changes that shouldn't go live immediately:

Step-by-Step Process:

  1. Update your local development branch: Start from the latest code.
    2. git checkout development  # Switches to the development branch
git pull origin development  # Fetches and merges the latest changes from remote development
  2. Create a feature branch (optional but recommended): For better organization, especially if working on multiple features. Name it descriptively, e.g., feature/new-login-system.
    3. git checkout -b feature/[feature-name]  # Creates and switches to a new feature branch
  3. Make your changes: Develop, test, and iterate.
  4. Commit your changes: Commit often with meaningful messages.
    5. git add .  # Stages all modified and new files
git commit -m "Add: [brief description]"  # Commits staged changes with a message
  5. Push the feature branch to remote: (For backup or collaboration).
    6. git push origin feature/[feature-name]  # Pushes the feature branch to the remote repository
  6. Merge feature into development: Once the feature is ready and tested.
    7. git checkout development  # Switches to the development branch
git merge feature/[feature-name]  # Merges feature branch changes into development
git push origin development  # Pushes updated development branch to remote
  7. Delete the feature branch: Clean up after merging.
    8. git branch -d feature/[feature-name]  # Deletes the local feature branch
git push origin --delete feature/[feature-name]  # Deletes the remote feature branch
  8. When ready to deploy: Merge development into main to trigger Render deployment. Do this only after thorough testing.
    9. git checkout main  # Switches to the main branch
git pull origin main  # Fetches and merges the latest changes from remote main
git merge development  # Merges development changes into main
git push origin main  # Pushes updated main branch to remote, triggering deployment

Best Practices for Development:

  • Use feature branches to isolate work and make pull requests if collaborating.
  • Regularly merge hotfixes or main into development to stay in sync: git merge main while on development.
  • Test extensively before merging to main to avoid breaking the live app.
  • Consider using Git tags for releases, e.g., git tag v1.0.1; git push origin v1.0.1.

General Tips

  • Resolve Conflicts: If merges cause conflicts, use git status (shows conflicted files) and edit files manually, then git add and git commit.
  • Undo Mistakes: Use git revert [commit-hash] to undo a commit or git reset carefully to rewind changes.
  • View Branch Status: git branch -a lists all branches; git log --graph --oneline --all shows a visual commit history.
  • Always back up your work by pushing to remote.

If you encounter issues, refer to the official Git documentation or tools like GitHub Desktop for a GUI alternative.
